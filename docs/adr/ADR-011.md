# Architectural Decision Record: ADR-011
**Title:** Generated Reports Strategy  
**Status:** Approved  
**Date:** 2026-06-22  

---

## Context
In our construction operations platform, several reporting structures are required to give executives high-level visibility over multi-currency tenders, active site compliance logs, milestone status percentages, and budget tracking parameters. 

Previously, some generated reports (like Single Paper Reports or certain dashboard analytics widgets) were modeled with persistent transactional tables or state structures inside legacy views. This created several critical issues:
1.  **Stale Indicators**: Storing transient dashboard metrics in database records risks desynchronization when underlying source transactions (Interim Payment Certificates, Variation Orders, Bids) get updated, approved, or deleted.
2.  **Schema Bloat**: Establishing standalone CRUD processes and database tables for read-only aggregation views introduces unnecessary database migrations, slow synchronization pipelines, and redundant index maintenance costs.
3.  **Audit Drift**: Hard-persisting derived metrics risks mismatching real historic audit trails if exchange rates or PMO offset rules are modified.

---

## Decision
All project reports, analytical metrics, and executive charts must be **generated dynamically on demand**. Real transaction schemas are the exclusive source of truth; analytics views assemble and compute relevant metrics at runtime inside secure calculation services rather than storing state in database records.

This strategy applies explicitly to:
1.  **Single Paper Report (SPR)**: Compiled dynamically by aggregating the baseline `Tender` details and relevant execution logs (`IPC`, `Claim`, `Variation Order`, `NOC`).
2.  **Dashboard KPIs**: Grand total values, critical count thresholds, and regional estimates computed on-the-fly inside `CalculationService`.
3.  **Executive summaries / Trends**: Real-time analytical statistics fetched and normalized dynamically.
4.  **Future analytics and visual widgets**: Any future charts, exportable sheets, or executive reports.

Reports are **never persisted** to standard relational tables or document collections unless explicitly requested by the client (for instance, to capture a signed "snapshot" PDF audit trail frozen in historic time).

---

## Consequences
*   **Data Integrity (Guaranteed)**: Dashboard KPIs and SPR sheets will always mirror actual real-time database records immediately upon rendering, eliminating state synchronicity bugs.
*   **Infrastructure Leanliness**: Reduces database storage footprints and limits structural index reads. No dedicated repositories or REST CRUD routes are needed for reports.
*   **Architectural Simplicity**: Simplifies the front-end rendering engines. View screens query live, calculated payloads directly from memory aggregates or lightweight service queries.
